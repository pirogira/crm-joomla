# CRM для Joomla 4/5

Компонент CRM с управлением стадиями компаний, журналом событий и discovery-формами.

---

## Архитектура

```
┌─────────────────────────────────────────────────────────────────┐
│  Joomla MVC (site)                                               │
│  ┌─────────────┐  ┌──────────────┐  ┌─────────────────────────┐  │
│  │ Controller  │→ │ Model        │→ │ View (карточка компании) │  │
│  │ (task)      │  │ CompanyModel │  │ default.php             │  │
│  └──────┬──────┘  └──────┬───────┘  └─────────────────────────┘  │
│         │                │                                       │
│         ▼                ▼                                       │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │ CrmCore                                                      │ │
│  │  • StageMachine (переходы стадий)                            │ │
│  │  • State-классы (ограничения по crm_events)                  │ │
│  │  • DatabaseConnector → #__companies, #__crm_events            │ │
│  └─────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
```

- **Слой данных**: `DatabaseConnector` + `DatabaseConnectorInterface` (для тестов)
- **Доменная логика**: `StageMachine` + State-паттерн (9 стадий, 3 бизнес-ограничения)
- **Представление**: Joomla view с индикатором стадии, кнопками действий, инструкцией и лентой событий

---

## Модель данных

| Таблица | Назначение |
|---------|------------|
| `#__companies` | Компании: id, name, current_stage, created_at |
| `#__crm_events` | Журнал событий: company_id, event_type, payload (JSON), created_at |
| `#__discovery_forms` | Discovery-анкеты: company_id, form_key, data (JSON), created_at |

**Индексы** (под нагрузку ~10k записей/день):
- `companies(current_stage, created_at)` — фильтр по стадии
- `crm_events(company_id, created_at)` — лента событий компании
- `crm_events(event_type, created_at)` — аналитика по типу

**Стадии** (порядок): Ice → Touched → Aware → Interested → Demo_planned → Demo_done → Committed → Customer → Activated

## Установка и использование

### Установка схемы

Выполните SQL из `sql/install.mysql.utf8mb4.sql` в вашей базе Joomla (через миграцию компонента или вручную).

### Использование в коде

Пример:

```php
use Vendor\Crm\DatabaseConnector;
use Vendor\Crm\CrmStage\CrmEventType;
use Vendor\Crm\CrmStage\Stage;
use Vendor\Crm\CrmStage\StageMachine;

$dbx = new DatabaseConnector();
$companyId = $dbx->createCompany('Acme LLC', Stage::ICE);

// События — единственный источник истины для ограничений переходов.
$dbx->addCrmEvent($companyId, CrmEventType::TALK_WITH_DECISION_MAKER, ['note' => 'созвон с ЛПР']);
$dbx->addCrmEvent($companyId, CrmEventType::DEMO_CONDUCTED, ['who' => 'PM', 'result' => 'ok']);

$sm = new StageMachine($dbx);
$sm->transition($companyId, Stage::TOUCHED);
$sm->transition($companyId, Stage::AWARE);         // разрешено только после 'разговор с ЛПР'
$sm->transition($companyId, Stage::INTERESTED);
$sm->transition($companyId, Stage::DEMO_PLANNED);  // запрещено, если ниже Aware
$sm->transition($companyId, Stage::DEMO_DONE);     // требует demo_conducted не старше 60 дней
$forms = $dbx->getDiscoveryFormsByCompany($companyId);
```

### Joomla view (карточка компании)

Шаблон карточки:

- `components/com_crm/tmpl/company/default.php`

View/Model для передачи данных в шаблон:

- `components/com_crm/src/View/Company/HtmlView.php`
- `components/com_crm/src/Model/CompanyModel.php`

Шаблон включает:

- индикатор текущей стадии
- кнопки действий, доступные только для текущей стадии
- блок "Инструкция/скрипт" по стадии
- ленту событий из `#__crm_events`

### AI-Workflow

**Используемые AI-инструменты**: Cursor (IDE с встроенным AI), чат для генерации кода и рефакторинга.

**Процесс**: разбиение на задачи (схема БД → connector → State-паттерн → view), формулировка промптов с явными ограничениями («crm_events — единственный источник истины»), проверка результата через unit-тесты и ручной прогон.

**Контроль качества**: явные требования в промптах против галлюцинаций; лицензии — MIT-совместимый код; безопасность — параметризованные запросы, без сырого SQL.

**Где AI дал выигрыш**: генерация DDL и индексов, CRUD-слой, State-классы, view-шаблон, unit-тесты, документация.

Этот проект собран итеративно в Cursor: сначала определили данные/нагрузку, затем зафиксировали доменные инварианты (ограничения переходов), после чего сгенерировали “вертикальный срез” (схема БД → доступ к данным → доменная логика → view).

#### 1) Как использовали Cursor для генерации бойлерплейта и рефакторинга

- **Быстрая генерация каркаса**: одним запросом получили DDL таблиц (`companies`, `crm_events`, `discovery_forms`) с `#__`-префиксом и базовыми индексами под частые запросы.
- **Сборка слоя доступа к данным**: сгенерирован `src/DatabaseConnector.php` с Query Builder Joomla (CRUD + целевые методы `hasEvent()` / `hasEventSince()` под проверки правил).
- **Выделение доменной логики**: вынесены стадии/правила в отдельный неймспейс `src/CrmStage/*` (константы стадий, состояния, фабрика, `StageMachine`).
- **Рефакторинг под “единственный источник истины”**: условия переходов привязаны к `#__crm_events`, чтобы не плодить флаги/поля в `companies` и не расходиться в данных.

#### 2) Как промпты помогли избежать галлюцинаций в логике ограничений

Ключевое требование “**используй `crm_events` как единственный источник истины**” было сформулировано явно и повторялось в задаче про State-паттерн. Это “заземлило” реализацию:

- **Переход в `Aware`**: разрешён только при наличии события `event_type = 'разговор с ЛПР'` (см. `CrmEventType::TALK_WITH_DECISION_MAKER` и проверку в `TouchedState`).
- **`Demo_done`**: разрешён только если найдено событие `event_type = 'demo_conducted'` за последние 60 дней (проверка через `hasEventSince()`).
- **`Demo_planned`**: запрещён, если текущая стадия ниже `Aware` (глобальная проверка перед переходом).

Такая постановка задачи не оставляет “места фантазии” (типа “разрешим демо по полю companies.demo_done_at”), потому что все факты должны быть подтверждены логом событий.

#### 3) Где AI дал наибольший выигрыш в скорости

- **Архитектура БД и индексы**: быстро собрали минимальную схему и индексацию под типовые выборки:
  - `companies(current_stage, created_at)` для фильтрации по стадии с сортировкой/диапазонами по времени
  - `crm_events(company_id, created_at)` для ленты событий компании
- **Вертикальная интеграция без “склейки руками”**: за короткое время появился рабочий путь “данные → доменная логика → view” (Joomla MVC view карточки компании).
- **Тестируемость правил**: сами ограничения описаны как детерминированные проверки по `crm_events`; это облегчает написание тестов (достаточно подготовить события и проверить `StageMachine->transition()`).

#### Масштабирование: журнал событий и минимизация блокировок (10k компаний/день)

- **Event log как источник истины**:
  - `#__crm_events` хранит неизменяемую историю (append-only), позволяя восстанавливать состояние и объяснять, почему стадия стала такой.
  - `#__companies.current_stage` остаётся “срезом” для быстрых списков/фильтров в UI (не заменяет историю).
- **Минимизация блокировок**:
  - запись событий — это `INSERT` в `crm_events` (минимальная конкуренция при корректных индексах, InnoDB справляется хорошо).
  - переход стадии — одиночный `UPDATE` по PK (`companies.id`) после успешных проверок (короткая блокировка строки).
  - проверки условий — это “быстрые” `SELECT 1 ... LIMIT 1` по индексам (`company_id, created_at`), без full scan.
- **Практические рекомендации**:
  - всегда добавляйте событие (например, “demo_conducted”) перед попыткой перехода, чтобы переходы были воспроизводимы из логов.
  - для тяжёлой аналитики по событиям используйте отдельные отчётные запросы/агрегации и не перегружайте карточку компании.

#### 4) AI-инструменты, процесс, контроль качества

- **Инструменты**: Cursor (IDE с AI), встроенный Copilot для автодополнения.
- **Процесс**: разбиение на задачи (схема → connector → State → view → тесты), промпты с явными ограничениями («crm_events — единственный источник истины»), проверка результата прогоном тестов и ручным просмотром кода.
- **Контроль качества**: явная формулировка инвариантов в промптах (снижение галлюцинаций), проверка сгенерированного SQL на соответствие Joomla-префиксу `#__`, лицензии — только MIT/стандартные библиотеки Joomla.
- **Выигрыш AI**: код (бойлерплейт, рефакторинг), тесты (FakeDatabaseConnector, 17 unit-тестов), документация (README, BUG_FIX_CYCLE), UX-тексты (инструкции по стадиям в language-файлах).
